# Research Ideation — 한계 극복 방안 도출 절차 (3인 연구팀)

3명의 연구자(`researcher-novelty`, `researcher-pragmatist`, `researcher-critic`)가 토론을 통해 합의안을 만드는 절차.

## 단계 1: 개별 초안 작성

세 연구자가 동일한 입력(`validation_report.md`)을 읽고 **독립적으로** 초안을 작성한다:

| 팀원 | 산출물 | 내용 |
|------|--------|------|
| novelty | `ideas_novelty.md` | 5개 이상의 후보 아이디어 (학술 근거, 수치 가설, 위험요소) |
| pragmatist | (대기) | novelty 초안 후 검토 시작 |
| critic | (대기) | 두 초안 후 비판 시작 |

> novelty가 먼저 제안, pragmatist와 critic은 응답형으로 시작. 초기 1라운드는 순차적.

## 단계 2: 토론 라운드 (반복)

`discussion_log.md`에 모두 함께 기록. 각 발언은 다음 형식:

```
[YYYY-MM-DD HH:MM] {팀원명}: {발언}
  대상: {idea-N} 또는 {previous-message-K}
  유형: 제안/비판/방어/수용/대안/질문
```

**토론 규칙:**
1. **모든 새 아이디어에 critic은 최소 1회 비판**한다 (게이트키퍼 역할)
2. pragmatist는 모든 아이디어에 **현장 적용 점수**를 매긴다
3. novelty는 비판에 응답하지 못한 아이디어를 **48시간 내 보강 또는 폐기**
4. 토론은 **최대 3라운드** (라운드 = 모든 팀원이 1회씩 발언)
5. 의사결정은 **다수결 아닌 합의 우선**, 합의 실패 시 "이견 보존"으로 표시

**팀 통신:**
- 직접 응답 → `SendMessage(to: 특정팀원)`
- 전체 공유 → 파일에 기록 (브로드캐스트 SendMessage는 비용 높음, 회피)

## 단계 3: 평가 매트릭스 작성

토론 후 모든 아이디어를 정량 평가:

```
| 아이디어 ID | 기대 효과 (정량) | 구현 난이도 | 데이터 요구 | 적용 확률 | 비판 잔존 | 종합 |
|------------|-----------------|------------|------------|----------|-----------|------|
| novelty-1  | recall +12%p    | 중          | MVTec AD   | 55%       | 2개 미해소 | B    |
```

**점수 산정 규칙** (확률적 항목은 서술 근거 필수):
- 기대 효과: 정량 수치 (가설 단계 명시)
- 적용 확률: % + "근거: (1)..., (2)..."

## 단계 4: 합의안 정리

`consolidated_proposals.md` 에 다음을 포함:

```
# 한계 극복 방안 합의서

## 1. 평가 매트릭스 (단계 3 결과)

## 2. 채택 제안 (Top-N)
- 채택 기준: 적용 확률 ≥ 50%, 비판 잔존 ≤ 1개
- 각 제안에 대해:
  - 핵심 메커니즘
  - 활용 오픈소스
  - 검증 실험 계획 (개발자에게 전달용)
  - 폐기 임계점 (어떤 결과가 나오면 중단)

## 3. 보류 제안 (재논의 필요)

## 4. 이견 보존
- 합의 실패 항목과 각 팀원의 입장

## 5. 잔존 리스크 (critic 작성)
```

## 단계 5: 사용자 의사결정 요청

오케스트레이터에게 알리고 사용자에게 다음을 제시:
- 합의서 요약
- 어떤 제안을 우선 구현할지 선택 요청
- 추가 토론 필요 시 라운드 추가 요청

## 후속 작업 (재실행)

이전 산출물이 존재하면:
1. `consolidated_proposals.md`를 모두 다시 읽음
2. 사용자 피드백 또는 새 발견을 반영
3. 토론 라운드를 추가 진행 (이전 라운드는 보존)

## 출력 위치 모음

```

├── ideas_novelty.md

├── critique_notes.md

```

## 품질 기준

- 모든 확률에 서술적 근거 첨부
- 모든 수치적 비교에 출처 명시
- 토론 로그가 비어 있으면 합의 무효 (실제 비판/검증이 있었는지가 핵심)
